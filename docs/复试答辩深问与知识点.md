# 复试答辩深问与知识点（本项目版）

## 1. 项目主线（一句话）
本项目把“图生文（结构化提示词）-> 文生图（参数可控生成）”打通，重点解决 **可控性、可复现性、风格化输出与工程可解释性**。

---

## 2. 老师高频深问（含答题框架）

### Q1：你的系统核心创新点是什么？
- 不是单纯做接口调用，而是把提示词拆成 `subject/scene/style/tech/negative/params` 六块结构化数据，再编译到真实生成链路。
- 可控参数（seed/steps/cfg/size/sampler）被显式透传到生成服务，形成“输入可解释 -> 输出可追溯”。

### Q2：为什么要做结构化提示词，不直接自然语言？
- 自然语言可读但难控制；结构化提示词可拆分责任，方便做参数优先级、风格注入和实验对照。

### Q3：结构化字段具体如何落地？
- 通过 `compile_structured_prompt` 编译为正向/负向 prompt，再与 mode/seed/steps/cfg/size 一起送入生成服务。

### Q4：参数优先级怎么设计？
- 采用：**显式表单参数 > JSON structured.params > 默认值**。
- 理由：用户在线调参应覆盖自动建议；默认值保证兼容旧请求。

### Q5：为什么要保留旧 JSON 兼容？
- 项目处于迭代期，前端和历史脚本不能一次性重写。兼容可降低回归风险。

### Q6：可复现性如何定义？
- `sd + 固定 seed` 记为 `strong`（同环境下高一致）。
- `proxy/gemini` 记为 `best_effort`（接口层可追踪，但不承诺像素级一致）。

### Q7：为什么云端模式难像素级复现？
- 云端模型可能存在版本热更新、后端采样策略不可见、服务侧随机项不可完全锁定。

### Q8：风格控制如何实现？
- 增加 `style_preset`（photoreal/anime/watercolor/cyberpunk/ink），在编译前做增量注入。
- 用户主体字段优先，避免风格词喧宾夺主。

### Q9：风格注入的风险是什么？
- 过强风格会导致主体漂移。缓解：预设仅增量词，并保留用户原字段优先级。

### Q10：如何证明“结构化可控”而不是口头描述？
- 返回 `meta.effective_params / prompt_source / style_applied / reproducibility`，前端直接展示。

### Q11：为什么要返回 effective_params？
- 便于实验记录和答辩展示“实际生效值”，避免“前端填了但后端没吃到”的黑箱问题。

### Q12：你如何处理非法参数？
- 对 `size/steps/cfg/seed` 做边界校验，非法直接 400，避免脏请求进入模型层。

### Q13：系统有哪些关键模式？
- `sd`（本地强可控）、`gemini`（云端）、`proxy`（远程代理）。
- 统一入口，模式差异在服务层分发。

### Q14：为什么要统一入口而不是分多个接口？
- 接口稳定、前端简单、方便做同条件对照实验。

### Q15：测试策略如何设计？
- `test_reverse.py` 验证图生文 schema 与提示词编译。
- `test_generate.py` 验证旧请求兼容、参数覆盖优先级、非法参数边界。

### Q16：如何做复现实验？
- 固定 prompt + mode=sd + seed 固定，多次生成对比 hash/视觉一致性。

### Q17：如果复现实验不稳定怎么办？
- 先锁环境（依赖版本、CUDA、模型权重、推理设备），再观察是否是服务端随机项或调度器差异。

### Q18：项目里最关键的工程取舍是什么？
- 取舍点：先实现“可解释可控链路”而非复杂 UI。
- 原因：复试更看重工程思路和可验证性。

### Q19：如何保证前后端字段一致？
- 后端 `GenerateMeta` 扩展后，同步更新前端 `types/reverse.ts`，避免隐式字段漂移。

### Q20：安全治理做了什么？
- 新增配置尽量走环境变量，减少硬编码敏感信息。
- 接口层做文件类型/大小限制和参数校验。

### Q21：为什么不把所有异常都吞掉？
- 调试期需要可诊断错误；边界错误给 400，内部错误给 500，便于定位。

### Q22：如何回答“你的项目深度在哪里”？
- 深度不在“调用了模型”，而在“建立了可控参数链路 + 可复现等级定义 + 风格策略 + 测试证据”。

### Q23：你的系统目前局限是什么？
- 云端模式复现性有限；sampler 在不同后端语义不一致；风格预设仍是规则驱动。

### Q24：下一步可做哪些研究增强？
- 加入自动指标（CLIPScore/美学分）做定量评估。
- 做风格强度系数与主体保真度权衡实验。

### Q25：论文实验如何分组更有说服力？
- A组：非结构化 prompt。
- B组：结构化 prompt（无风格预设）。
- C组：结构化 + 风格预设。
- 比较可复现性、风格一致性、主观质量。

### Q26：如何避免“实验偶然性”？
- 每组至少多 seed、多样本，报告均值与失败样例。

### Q27：如果老师问“你做的不是前端堆界面吗”？
- 回答重点：前端只做控制面板；核心是后端参数归一化、编译复用与服务透传。

### Q28：为什么要保留 strict_json？
- 作为实验对照：比较“JSON直传”与“提示词封装”在不同后端下的行为差异。

### Q29：如何证明系统具备工程可维护性？
- schema 明确、接口兼容、模式统一入口、测试覆盖关键边界。

### Q30：复试现场 demo 最小流程？
1. 上传图片跑 `/reverse` 得到 structured。
2. 用 JSON 上传 `/generate`，切换 mode/style。
3. 锁定 seed 在 sd 模式重复生成，展示复现。
4. 展示 `effective_params/reproducibility` 做可解释说明。

## 5. 复试回答技巧（30秒版）
- 先说问题：以前“有结构化数据但没打通到生成参数”。
- 再说方案：归一化层 + 编译复用 + 参数透传 + 风格预设 + 元信息回传。
- 最后说证据：接口响应可解释字段 + 自动化测试 + 可复现实验记录。
