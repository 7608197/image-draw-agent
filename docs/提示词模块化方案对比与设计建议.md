# 提示词模块化方案对比与设计建议

## 1. 当前项目提示词结构梳理

结合当前代码，项目已经不是“纯字符串 prompt”方案，而是一个 **结构化提示词 + 编译输出** 的体系。

### 1.1 当前后端结构
当前核心结构定义在：`backend/schemas.py`

项目已经将提示词拆成 6 个 block：
- `subject`
- `scene`
- `style`
- `tech`
- `negative`
- `params`

其中：

#### subject
负责主体内容：
- `entities`
- `label`
- `attributes`
- `count`
- `weight`

#### scene
负责场景信息：
- `environment`
- `background`
- `time_weather`
- `composition`

#### style
负责风格表达：
- `medium`
- `artist_style`
- `aesthetic`
- `quality`

#### tech
负责技术性细节：
- `lighting`
- `camera`
- `color_tone`
- `render`

#### negative
负责负向提示词：
- `terms`
- `severity`
- `term_weights`

#### params
负责生成参数：
- `size`
- `steps`
- `cfg`
- `sampler`
- `seed`

---

### 1.2 当前 prompt 编译方式
当前正负 prompt 的编译逻辑主要在：
- `backend/services/reverse_service.py:235`
- `frontend/src/utils/prompt.ts:15`

编译顺序基本是：
1. `subject`
2. `scene`
3. `style`
4. `tech`
5. `negative` 单独编译

也就是说，你当前项目已经具备：
- **结构化存储**
- **线性编译输出**
- **负向词单独管理**
- **参数与语义分离**

这是一个比较扎实的基础。

---

### 1.3 当前 generate 端的增强机制
当前 `/generate` 入口在：`backend/main.py:318`

它已经支持：
- 上传 JSON 直接生成
- `structured` 输入优先编译
- `style_preset` 对 `structured` 做增量合并
- 表单参数覆盖 `structured.params`
- legacy / json_spec 兜底

其中你现在最接近“模块化提示词”的实现，是：
- `backend/main.py:49` 的 `STYLE_PRESETS`
- `backend/main.py:199` 的 `_apply_style_preset_to_structured`

目前的 `style_preset` 本质上是：
- 对 `style`
- 对 `tech`
- 对 `negative`
进行一次 **预设式增量注入**。

这已经体现出“按需加载一组细节提示词”的雏形。

---

## 2. 当前方案的优点

你的现有结构相比纯 prompt 拼接，有几个明显优点：

### 2.1 可解释性强
每类提示词有明确归属，便于前端展示，也便于答辩说明。

### 2.2 前后端一致
后端 `schemas.py` 与前端 `types/reverse.ts` 基本对齐，数据契约比较清晰。

### 2.3 适合 reverse -> generate 闭环
图生文返回的 `structured` 可以直接进入文生图编译链路，这点很适合你的毕业设计主题。

### 2.4 已有“风格模块”的初步能力
`STYLE_PRESETS` 已经说明系统不是只能吃完整 prompt，也支持预设块注入。

---

## 3. 当前方案的不足

虽然当前结构已经比较好，但如果目标是你说的：
- 用户追加细节要求
- 对细节提示词模块化修改
- 对模糊需求进行默认补全
- 支持迁移与重绘

那么目前结构还存在几个明显缺口。

### 3.1 缺少“任务层”
当前结构默认是在做通用生成，但没有显式区分：
- 文生图
- 图生图
- 重绘
- 风格迁移
- 局部编辑

这会导致同样的结构化提示词，在不同任务里缺少不同的装配策略。

例如：
- 文生图更关注完整语义表达
- 重绘更关注“保留什么、修改什么”
- 迁移更关注“替换风格层，而不是重建主体层”

当前 `StructuredPrompt` 中还没有承载这一层信息。

---

### 3.2 细节词仍然偏“平铺列表”
当前 `style.medium`、`tech.lighting`、`tech.color_tone` 等虽然是分块的，但块内部仍然是简单字符串列表。

这意味着系统目前知道：
- 某个词属于 lighting
- 某个词属于 color_tone

但还不知道：
- 它是哪个模块名触发的
- 它的默认值是什么
- 它和哪个模块互斥
- 用户是显式指定，还是系统自动补全
- 哪个词是核心词，哪个词只是增强词

也就是说，现在是“结构化字段”，但还不是“结构化模块”。

---

### 3.3 缺少模糊需求映射层
你想做的是：
- 用户说“更梦幻一点”
- 用户说“更高级一点”
- 用户说“像海报”

系统就能自动映射到默认细节模块。

但当前代码里没有这样一层：
- 用户自然语言意图
- 到模块名
- 再到具体 prompt 片段

现在 `style_preset` 还是显式选项式的，不是从自然语言自动触发的。

---

### 3.4 缺少冲突处理机制
当前 `_apply_style_preset_to_structured` 主要做的是去重合并，而不是语义冲突消解。

例如如果同时出现：
- `minimal background`
- `rich detailed background`

当前更像是两个都加进去，而不是保留其中一个。

如果后续细节模块越来越多，prompt 很容易出现：
- 同类词堆叠
- 风格打架
- 约束冲突

---

### 3.5 缺少“保留/修改”约束层
对于迁移和重绘，最关键的不是再生成一遍 prompt，而是控制：
- 保留主体
- 保留姿态
- 保留构图
- 替换风格
- 只增强背景

当前结构没有显式表达：
- `keep`
- `change`
- `edit_strength`
- `region_scope`

所以它目前更适合“重生成”，不够适合“可控编辑”。

---

## 4. 我提出的方案与当前项目的对比

## 4.1 相同点
我的方案和你当前项目，并不是冲突关系，而是递进关系。

两者共同点：
1. 都不是直接依赖单条字符串 prompt
2. 都强调先结构化，再编译输出
3. 都把主体、场景、风格、技术、负向词、参数分开治理
4. 都适合做图生文 -> 文生图闭环

所以可以说：
**你当前项目已经完成了“结构化提示词 1.0”，而我前面提的方案更像是“模块化提示词 2.0”。**

---

## 4.2 差异点

### 当前项目更像：字段分块
例如：
- `tech.lighting = ["soft light"]`
- `style.aesthetic = ["cinematic"]`

优势是简单、清晰、容易展示。

### 我提出的方案更像：模块装配
例如：
- `lighting.soft_backlight`
- `mood.dreamy`
- `composition.poster`
- `background.rich_scene`

每个模块不仅有 prompt 文本，还应该有：
- 模块名
- 类别
- 触发词
- 默认值
- 优先级
- 互斥组
- 是否自动补全

也就是说：

| 对比项 | 当前项目 | 建议方案 |
|---|---|---|
| 组织方式 | block 字段 | block + module |
| 数据粒度 | 词条列表 | 命名模块 |
| 风格增强 | style preset | 可细分模块按需加载 |
| 模糊需求支持 | 弱 | 强 |
| 冲突消解 | 基本去重 | 分类覆盖/互斥处理 |
| 迁移/重绘支持 | 间接支持 | 显式支持 |

---

## 5. 推荐的演进思路

我不建议你推翻当前结构重做。

更适合的方式是：
**保留现有 6-block 结构，把“模块层”加在其上，而不是替代它。**

这样可以最大化复用你现有：
- schema
- 前端展示
- reverse 输出
- compile 逻辑
- generate 接口

---

## 5.1 演进方向一：保留 6-block，新增任务层
建议新增一个任务控制对象，例如：

```json
{
  "task": {
    "mode": "text_to_image",
    "source": "user_input"
  }
}
```

后续可以扩展为：
- `text_to_image`
- `image_to_image`
- `redraw`
- `style_transfer`
- `local_edit`

作用：
- 决定 prompt 装配策略
- 决定默认模块加载规则
- 决定哪些字段是保留项、哪些是可编辑项

---

## 5.2 演进方向二：在 block 内引入“模块元信息”
当前 block 内是字符串列表，可以升级为“模块实例列表”，或者额外加一个模块清单。

例如保守做法：
保留现有字段不变，再新增一个 `modules` 字段：

```json
{
  "modules": [
    {
      "name": "soft_backlight",
      "category": "lighting",
      "prompt_text": "soft backlight, gentle rim light",
      "source": "default",
      "enabled": true
    },
    {
      "name": "dreamy_mood",
      "category": "mood",
      "prompt_text": "dreamy atmosphere, soft haze, floating particles",
      "source": "user_request",
      "enabled": true
    }
  ]
}
```

这样做的好处：
- 不破坏你现有前端结构
- 旧编译逻辑还能继续工作
- 新逻辑可以基于 `modules` 进行更强控制

---

## 5.3 演进方向三：把 style preset 拆成更细粒度模块库
你当前的 `STYLE_PRESETS` 已经是很好的起点。

但它目前粒度偏大，例如：
- `anime`
- `watercolor`
- `cyberpunk`

建议未来拆成：
- 风格类：`anime_render`, `watercolor_texture`, `oil_paint_brush`
- 光照类：`soft_light`, `rim_light`, `dramatic_contrast`
- 配色类：`pastel_tone`, `neon_cyan_magenta`, `warm_gold`
- 构图类：`poster_layout`, `close_up`, `wide_shot`
- 背景类：`minimal_background`, `rich_background`
- 质量类：`ultra_detail`, `clean_edges`

也就是：
从“整包风格预设”逐步演化到“可组合细节模块库”。

---

## 5.4 演进方向四：增加模糊词映射表
这是你想实现“像 skills 一样按需加载”的关键。

例如建立一个映射层：

```json
{
  "dreamy": ["soft_light", "dreamy_mood", "pastel_tone"],
  "高级感": ["cinematic_lighting", "clean_composition", "premium_texture"],
  "海报感": ["poster_layout", "strong_focus", "title_safe_space"]
}
```

流程变成：
1. 解析用户追加要求
2. 命中触发词
3. 找到对应模块
4. 未命中时加载默认模块
5. 将模块编译回现有 `style/tech/scene` 等 block

这个设计最适合你的“细节词按需加载”。

---

## 5.5 演进方向五：增加约束层，服务迁移与重绘
建议增加一个编辑约束对象，例如：

```json
{
  "edit_control": {
    "keep": ["subject", "pose", "composition"],
    "change": ["style", "lighting", "background"],
    "strength": 0.65
  }
}
```

它的作用不是直接给模型，而是先参与 prompt 装配策略。

比如：
- 如果 `keep` 包含 `composition`，就避免覆盖构图模块
- 如果 `change` 包含 `style`，就允许替换 style 类模块
- 如果 `strength` 较低，就减少高侵入性细节模块

这一步对“迁移与重绘画”尤其重要。

---

## 6. 最适合你当前项目的落地方式

结合你现在的工程状态，我认为最合适的不是大改 schema，而是分三步渐进。

### 第一步：保留当前 schema，不动主链路
保留：
- `StructuredPrompt`
- `compile_structured_prompt`
- 前端展示结构
- `/reverse` 输出结构

因为这些已经能支撑答辩与演示。

---

### 第二步：新增“模块库 + 映射层”
新增一个模块库定义，例如放在后端配置中：
- module name
- category
- trigger words
- default prompt text
- conflict group
- priority

然后增加一个解析过程：
- 从用户追加要求中识别命中的模块
- 把模块内容回填到现有 `scene/style/tech/negative` block

这样你现有接口几乎不需要推翻。

---

### 第三步：新增“编辑约束层”服务迁移/重绘
后续如果你要做：
- 参考图重绘
- 风格迁移
- 局部编辑

再加：
- `task`
- `edit_control`

这一步是面向下一阶段功能扩展，不一定要在当前版本全部做完。

---

## 7. 一个适合论文/答辩的表述方式

你可以把当前系统和未来演进表达成两层：

### 当前已实现
本系统已实现：
- 图像内容的结构化提示词抽取
- 提示词按主体/场景/风格/技术/负向/参数六块组织
- 从结构化数据到生成 prompt 的自动编译
- 风格预设对结构化提示词的增量注入

### 下一步优化方向
在现有结构化提示词基础上，进一步引入：
- 细节提示词模块库
- 模糊语义到模块的映射机制
- 冲突消解与默认补全策略
- 面向迁移与重绘的保留/修改约束层

这样可以实现从“结构化提示词”向“模块化可控提示词装配”的升级。

## 8. 最终结论

### 8.1 对当前项目的判断
你的项目当前提示词体系已经具备较好的工程基础，尤其适合毕业设计展示，因为它已经完成了：
- 结构化
- 可编译
- 可解释
- 可闭环

这不是一个从零开始的问题，而是一个“如何在已有结构上继续升级”的问题。

### 8.2 对我方案的判断
我前面提出的方案，不应该替换你现有设计，而应该作为你当前系统的 **下一层增强机制**。

最合理的关系是：

- 当前系统：**结构化提示词层**
- 下一阶段：**模块化细节装配层**
- 再下一阶段：**迁移/重绘的可控编辑层**

### 8.3 一句话总结
**你现在已经做到了“把 prompt 拆成结构”，下一步最值得做的是“把细节词做成可按需加载、可覆盖、可默认补全的模块”。**

这条路线和你最开始的想法是一致的，而且与现有工程兼容性很好。
